### Reflection - Module 03
1. Explain what principles you apply to your project!
- Single Responsibility Principle (SRP): Prinsip ini diterapkan dengan pemisahan tanggung jawab yang jelas pada setiap kelas. CarReadController hanya bertanggung jawab untuk menangani request berupa pembacaan data Car (seperti menampilkan daftar dan halaman edit), sementara CarWriteController secara terpisah menangani semua operasi yang mengubah data (membuat, memperbarui, menghapus). Selain itu, pada lapisan repository, saya melakukan pemisahan antara interface untuk Read dan Write.
- Open/Closed Principle (OCP): Prinsip ini diterapkan dengan penggunaan interface yang terpisah pisah. Contohnya, CarQueryService dan CarCommandService adalah dua interface yang berbeda untuk mendefinisikan operasi untuk membaca dan menulis data mobil. Class-class controller (CarReadController dan CarWriteController) bergantung pada interface ini, bukan pada implementasi konkritnya. Jika kita ingin menambahkan perilaku baru misalnya logging pembuatan mobil, kita bisa membuat implementasi baru dari CarCommandService tanpa mengubah kode di CarWriteController. Dengan begitu, controller terbuka untuk ekstensi melalui implemetasi service baru, tetapi tertutup thp modifikasi.
- Liskov Substitution Principle (LSP): Prinsip ini sudah saya terapkan pada service dan repository. Interface seperti CarCommandService mengharapkan objek bertipe Car. Controller hanya tahu interface CarCOmmandService. Controller bisa diberi implementasi apapun yang mengimplementasi interface tersebut. Baik CarCommandServiceImpl maupun CarCommandServiceWithLogging bisa saling menggantikan tanpa merusak program. Mtehod create() pun sama sama bisa digunakan untuk membuat car seperti biasa.
- Interface Segregation Principle (ISP): Prinsip ini diterapkan dengan memisahkan interface besar menjadi interface yang lebih kecil dan spesifik. Contohnya, pada lapisan repository, terdapat CarReadRepository yang hanya berisi metode untuk membaca data (findById, findAll), dan CarWriteRepository yang hanya berisi metode untuk menulis data (create, update, delete). Demikian pula pada lapisan service, ada CarQueryService dan CarCommandService. 
- Dependency Inversion Principle (DIP): Prinsip ini saya terapkan dengan membuat controller tidak bergantung pada low-level class. High-level module (seperti ProductReadController) tidak bergantung langsung pada low-level module (seperti InMemoryProductRepository). Sebaliknya, keduanya bergantung pada abstraksi. ProductReadController bergantung pada interface ProductQueryService, dan ProductQueryServiceImpl (sebagai implementasi service) bergantung pada interface ProductReadRepository. Low-level module InMemoryProductRepository juga bergantung pada interface ProductReadRepository dengan mengimplementasikannya. Dengan sistem seperti ini, membuat sistem menjadi sangat fleksibel dan mudah untuk mengganti implementasi di masa depan.
2. Explain the advantages of applying SOLID principles to your project with examples.
Penerapan SOLID menguntungkan untuk pemeliharaan dan pengembangan jangka panjang. Sebagai contoh, karena kita menerapkan SRP dan DIP, jika kita ingin mengganti penyimpanan data dari InMemoryCarRepository ke database SQL, kita cukup membuat DbSqlCarRepository dan mengimplementasikan CarReadRepository dan CarWriteRepository. Kita kemudian tinggal mengonfigurasi dependency injection Spring untuk menggunakan implementasi baru ini. Kelas ProductQueryServiceImpl dan ProductWriteController tidak perlu diubah sama sekali karena mereka hanya bergantung pada interface (abstraksi). Hal ini menunjukkan kode kita yang loosely coupled.
3. Explain the disadvantages of not applying SOLID principles to your project with examples.
Jika SOLID tidak diterapkan, sistem akan sulit untuk dikembangkan dan diamintain
Jika prinsip SOLID tidak diterapkan, proyek ini akan sulit untuk dikembangkan dan di-extend. Sebagai contoh, jika kita tidak menerapkan ISP dan malah membuat satu interface ProductRepository yang berisi semua method CRUD, maka ProductQueryServiceImpl akan terpaksa bergantung pada interface tersebut, meskipun ia hanya membutuhkan method findAll dan findById. Akibatnya, jika suatu saat kita mengubah method delete di interface (misalnya, mengubah parameternya), kelas ProductQueryServiceImpl yang tidak peduli dengan method delete tersebut tetap harus dikompilasi ulang, meskipun tidak ada perubahan logika di dalamnya.
